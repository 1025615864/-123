# docs 目录清单与归档策略

更新时间：2026-01-08

本文档用于统一 `docs/` 的目录结构与归档规则，避免根目录文档堆叠，并确保“当前有效文档”和“历史归档文档”可追溯。

## 1. 目录结构（约定）

- `docs/README.md`
  - 本文件，作为 docs 入口
- `docs/PRD.md` / `docs/TECH_SPEC.md` / `docs/API_DESIGN.md` / `docs/DATABASE.md` / `docs/CHANGELOG.md`
  - 面向 AI 编程的“当前有效”文档口径（需求/技术/接口/数据/变更）
- `docs/_archive/`
  - 历史归档（阶段性交付/验收材料/过程文档等，仅供追溯）

## 2. 当前有效文档清单（入口推荐顺序）

- `docs/PRD.md`
  - 产品需求口径（功能范围/角色/验收点）
- `docs/TECH_SPEC.md`
  - 技术栈、项目结构、环境变量与工程约束
- `docs/API_DESIGN.md`
  - API 模块边界与关键接口汇总（以 Swagger 为权威）
- `docs/DATABASE.md`
  - 数据库表结构口径（以 SQLAlchemy models 为权威）
- `docs/CHANGELOG.md`
  - 对外可感知变更记录

历史交付/验收材料已归档：

- `docs/_archive/`

## 3. 归档规则

- `docs/` 根目录原则上只保留本 `README.md`（入口），其余文档按类型进入对应子目录。
- 当文档满足以下任一条件时，应移动到 `docs/_archive/`：
  - 已完成阶段性使命（如一次性分析报告、旧版方案、历史复盘）
  - 已被新版文档替代（保留历史可追溯性）
  - 内容不再适用于当前主线（避免误导）

## 4. 命名与维护约定

- 文件名尽量语义化、中文优先（便于甲方阅读），必要时可保留英文（如报告类文档已形成惯例）。
- 关键入口文档（交付总览、验收清单、执行计划）应维护“更新时间”。
- 若文档需要引用图片/附件，统一放置到 `docs/assets/` 并使用相对路径引用。
